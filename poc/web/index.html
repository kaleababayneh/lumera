The flywheel

Agent buys credits → Publisher lists a tool with a bond → Router locks payment → SuperNode proves the work (BLAKE3(input,model,output)) → Settlement pays the publisher only against that proof → reputation is earned from real receipts → a bad call is disputed, the bond is slashed, reputation erodes.

Run the loop · step by step

1
Agent buys credits — swap LUME → LAC
The agent converts the native LUME token into LAC spending credits.
2
Publisher lists a tool — register + bond
Registers pubtool and escrows a 2,000,000 ulume bond (skin-in-the-game).
3
Router locks payment — quote → lock
Locks 1,000,000 LAC for one call; returns a lock_id.
4
SuperNode proves the work — Proof-of-Service
An active SuperNode anchors receipt_id = pos1<BLAKE3(input,model,output)> on-chain.
5
Settlement — gated on the proof
Pays the publisher only if the receipt verifies for this tool & lock.
6
Reputation — earn a verification badge
The tool is scored from its on-chain metrics into a tiered badge (incentives).
7
A bad call is disputed — challenge the receipt
The challenger escrows a stake and locks an equal slice of the publisher's bond. Settlement freezes.
8
Dispute upheld — the bond is slashed
An adjudicator (active SuperNode, disjoint from both parties) upholds it → bond slashed, restitution-routed (5% burn / 85% insurance / 10% treasury). Re-run step 6 to watch reputation erode.

What this proves

Verifiable Execution → Economic Coordination. Settlement is contingent on a SuperNode-attested proof bound to the exact lock, so the agent only ever pays for verified work — and a receipt can't be replayed against another lock.

The trust graph then self-reinforces: real receipts raise reputation (a tiered badge), while an upheld dispute slashes the publisher's bond (restitution-routed) and erodes that reputation. Every primitive here is real and on-chain.
